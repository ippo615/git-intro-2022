Git File Lifecycle

Staged vs Unstaged

Tracked vs Untracked

Modified vs Unmodified

Staging

All files are unstaged by default.

To stage a file:

git add FILENAME

To unstage a staged file:

git reset FILENAME

Create a file

							Andrew@DESKTOP-45FU1JG MINGW64 ~/Downloads/sample-project (master)
							$ echo "Hello world!" > readme

Check Status

							Andrew@DESKTOP-45FU1JG MINGW64 ~/Downloads/sample-project (master)
							$ git status
							On branch master
							No commits yet
							Untracked files:
							  (use "git add FILE..." to include in what will be committed)
								
							    readme

Stage readme

							Andrew@DESKTOP-45FU1JG MINGW64 ~/Downloads/sample-project (master)
							$ git add readme
						
							Andrew@DESKTOP-45FU1JG MINGW64 ~/Downloads/sample-project (master)
							$ git status
							On branch master
							No commits yet
							Changes to be committed:
							  (use "git rm --cached FILE..." to unstage)
							
							    new file:  readme

Unstage readme

							Andrew@DESKTOP-45FU1JG MINGW64 ~/Downloads/sample-project (master)
							$ git reset readme
						
							Andrew@DESKTOP-45FU1JG MINGW64 ~/Downloads/sample-project (master)
							$ git status
							On branch master
							No commits yet
							Untracked files:
							  (use "git add ..." to include in what will be committed)
							
							    readme
							
							nothing added to commit but untracked files present (use "git add" to track)

Committing

After staging all your changes you should commit them.

git commit -m 'A commit message'

Stage readme

							$ git add readme
						
							Andrew@DESKTOP-45FU1JG MINGW64 ~/Downloads/sample-project (master)
							$ git status
							On branch master
							No commits yet
							Changes to be committed:
							  (use "git rm --cached FILE..." to unstage)
							
							    new file:  readme

Commit Changes

Tracking

Git automatically tracks committed files once you add and commit them.

Git will tell you if a tracked file has changed when you run git status.

Part 4: Working with a Remote Repository

You begin working with a remote repository by clone-ing it to your local machine.

Once the repo has been cloned you can edit things and commit them exactly as you would for a local repository.

You pull changes from the remote to your local copy to work with the newest code.

You push changes from your local copy to the remote to share your changes.

Cloning

git clone REPO_URL
